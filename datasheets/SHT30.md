# SHT30 (SHT3x-DIS Family) — Complete I²C Driver and Register/Command Reference

## Purpose

This document is a **self-contained implementation reference** for building a complete **I²C driver** for the **Sensirion SHT30-DIS** humidity and temperature sensor.

It is intended to be sufficient on its own for:

- autogenerated driver creation
- register/command-level firmware implementation
- wiring and integration questions
- timing and reset handling
- checksum validation
- conversion of raw outputs to physical units

Do not assume access to any external datasheet or source material.

---

## Scope and Device Identity

The **SHT30-DIS** is part of the **SHT3x-DIS** family. The family includes SHT30, SHT31, and SHT35 variants. The **I²C interface, command set, timing model, and status/heater/reset behavior are shared across the SHT3x-DIS family**. Accuracy specifications differ by variant.

This document targets **SHT30-DIS** specifically, while retaining shared SHT3x-DIS protocol details that are required to implement a full SHT30 driver.

---

## Overview

The SHT30-DIS is a fully calibrated digital **relative humidity** and **temperature** sensor with:

- factory calibration
- linearized digital output
- temperature-compensated and supply-compensated output
- I²C interface up to **1 MHz**
- two selectable I²C addresses
- single-shot measurement mode
- periodic measurement mode
- accelerated response time (**ART**) mode
- internal heater
- status register
- reset support through:
  - soft reset command
  - nRESET pin
  - I²C general call reset
  - interface reset sequence

## Pins and Wiring Constraints

## Pin assignment

| Pin | Name | Function / constraint |
|---|---|---|
| 1 | `SDA` | serial data, input/output |
| 2 | `ADDR` | address select input; connect to logic high or low; **must not float** |
| 3 | `ALERT` | alert output; **must be left floating if unused** |
| 4 | `SCL` | serial clock |
| 5 | `VDD` | supply voltage |
| 6 | `nRESET` | active-low reset input |
| 7 | `R` | no electrical function; connect to `VSS` |
| 8 | `VSS` | ground |

## Power and PCB wiring constraints

- decouple `VDD` to `VSS` with **100 nF** placed as close to the sensor as possible
- the center die pad is electrically connected to **VSS**
- the die pad should be soldered to the PCB for mechanical reasons
- pin 7 (`R`) must be connected to `VSS`

## `ADDR` pin rules

- `ADDR` selects one of two I²C addresses
- it may be changed dynamically during operation
- the only constraint is that the `ADDR` level must remain constant from the I²C START condition until the communication finishes
- `ADDR` **must not be left floating**

## `ALERT` pin rules

- output reflects alarm conditions relative to programmable limits
- if unused, **leave floating**
- the pin switches high when alert conditions are met
- self-heating may occur if significant current flows through the alert output
- using it to drive a transistor is recommended if load current would otherwise be significant

## `nRESET` pin rules

- active low
- internal pull-up to `VDD`, typical **50 kΩ**
- if unused, it is recommended to:
  - leave it floating, or
  - connect it to `VDD` through a series resistor `>= 2 kΩ`
- a low pulse of at least **1 µs** triggers a reset

---

## I²C Addresses

The SHT30 has two 7-bit I²C addresses:

| Condition | 7-bit address |
|---|---:|
| `ADDR` tied low | `0x44` |
| `ADDR` tied high | `0x45` |

### Addressing rules

- only the **7 MSBs** constitute the I²C address
- the read/write bit is the LSB appended by the bus transaction
- the address can be switched dynamically through `ADDR`, as long as it stays constant during an active transfer

---

## Protocol Model

## Address space and framing

- all commands and data are mapped to a **16-bit command/address space**
- commands are transmitted as **16-bit words**
- read and write data words are protected with **CRC-8**
- command words already include an internal **3-bit command checksum** in the command encoding
- every data word sent to or received from the sensor is followed by an **8-bit CRC**

## CRC requirements

### Write direction
In write direction, the checksum is **mandatory** whenever data words are written. The sensor only accepts data if it is followed by the correct checksum.

### Read direction
In read direction, it is up to the host to read and validate the checksum bytes. The sensor transmits them after each data word.

### Command spacing
After sending a command, wait at least **1 ms** before sending another command.

## Complete Command Set

All commands are **16-bit values**, transmitted MSB first.

## Single-shot measurement commands

These commands trigger one humidity/temperature measurement pair.

### With clock stretching enabled

| Repeatability | Command |
|---|---:|
| High | `0x2C06` |
| Medium | `0x2C0D` |
| Low | `0x2C10` |

### With clock stretching disabled

| Repeatability | Command |
|---|---:|
| High | `0x2400` |
| Medium | `0x240B` |
| Low | `0x2416` |

### Single-shot mode behavior

- one issued command triggers one measurement pair
- each measurement pair contains:
  1. temperature
  2. humidity
- each result is 16-bit and followed by an 8-bit CRC

## Periodic measurement commands

These commands start periodic acquisition.

| mps | High | Medium | Low |
|---:|---:|---:|---:|
| 0.5 | `0x2032` | `0x2024` | `0x202F` |
| 1 | `0x2130` | `0x2126` | `0x212D` |
| 2 | `0x2236` | `0x2220` | `0x222B` |
| 4 | `0x2334` | `0x2322` | `0x2329` |
| 10 | `0x2737` | `0x2721` | `0x272A` |

### Periodic mode behavior

- one issued command starts a stream of data pairs
- each data pair contains:
  1. temperature
  2. humidity
- clock stretching is **not selectable** in this mode
- at the highest mps setting, **self-heating may occur**

## Fetch data command

| Function | Command |
|---|---:|
| Fetch data | `0xE000` |

### Fetch behavior

- used to read the latest measurement in periodic mode
- if no measurement data is present, the sensor responds to the read header with **NACK**
- after fetch data is issued and the data is read, the data memory is cleared, so no measurement remains present until a new one is produced

## ART command

| Function | Command |
|---|---:|
| Periodic measurement with ART | `0x2B32` |

### ART behavior

- ART means **accelerated response time**
- after issuing this command, the sensor starts acquiring data at **4 Hz**
- ART follows the same general model as periodic mode:
  - use fetch data to read results
  - use break to stop it

## Break command

| Function | Command |
|---|---:|
| Break / stop periodic mode | `0x3093` |

### Break behavior

- stops periodic acquisition mode
- recommended before sending another command, except fetch data
- aborts any ongoing measurement
- returns the sensor to single-shot mode
- transition takes **1 ms**

## Reset-related commands

| Function | Command |
|---|---:|
| Soft reset | `0x30A2` |
| General call reset | address `0x00`, second byte `0x06` |

## Heater commands

| Function | Command |
|---|---:|
| Heater enable | `0x306D` |
| Heater disable | `0x3066` |

## Status register commands

| Function | Command |
|---|---:|
| Read status register | `0xF32D` |
| Clear status register | `0x3041` |

## Measurement Operation

## Power-up and idle state

After `VDD` reaches the power-up threshold `VPOR`, the sensor needs `tPU` to enter idle state. Once in idle state, it is ready to receive commands.

When powered but not measuring or communicating, the sensor automatically enters idle state for energy saving.

## Starting a measurement

A measurement sequence consists of:

1. START
2. I²C write header
3. 16-bit measurement command
4. ACKs from the sensor

Once the command is acknowledged, the sensor begins measuring humidity and temperature.

## Readout order

For all measurement result transfers:

1. **temperature** MSB
2. **temperature** LSB
3. **temperature CRC**
4. **humidity** MSB
5. **humidity** LSB
6. **humidity CRC**

Temperature is always sent first, humidity second.

---

## Single-Shot Readout Behavior

After a single-shot measurement command:

### No clock stretching mode
- if the host tries to read before data is ready, the sensor responds to the read header with **NACK**

### Clock stretching mode
- the sensor ACKs the read header
- then it holds `SCL` low until the measurement is complete
- after completion it releases `SCL` and transmits the results

### Partial read behavior
- the master may abort with `NACK` after any byte if subsequent data is not needed
- if reading humidity and temperature but not validating CRC, Sensirion recommends still reading the temperature CRC byte, then reading the two humidity bytes, and aborting before the humidity CRC if desired

---

## Periodic Readout Behavior

In periodic mode:

- measurement continues automatically
- the host must issue **fetch data** (`0xE000`) to retrieve the most recent pair
- if no new measurement is available, the read header receives **NACK**
- after the fetch readout, the data memory is cleared

---

## Reset Behavior

The device supports several reset paths.

## Interface reset sequence

If communication is lost, the serial interface may be reset by:

1. keep `SDA` high
2. toggle `SCL` **nine or more times**
3. issue a transmission START before the next command

This resets the **interface only**. The status register preserves its content.

## Soft reset

Send command:

```text
0x30A2
```

Behavior:
- resets the system controller
- reloads calibration data from memory
- valid when the sensor is in idle state

## Reset through general call

Send:

- general call address byte: `0x00`
- second byte: `0x06`

Behavior:
- functionally identical to using `nRESET`
- **not device-specific**
- all devices on the bus that support general call reset may reset
- only works when the sensor can process I²C commands

## Reset through `nRESET` pin

- pull `nRESET` low for at least **1 µs**
- this generates a reset similar to a hard reset

## Hard reset

- power `VDD` off and on again
- to avoid powering the sensor through ESD diodes, voltages on `SDA`, `SCL`, and `ADDR` must also be removed

## Calibration data reload note

The datasheet states that the sensor reloads calibration data **prior to every measurement by default**.

## Heater

The SHT30 includes an internal heater.

### Purpose
- intended for **plausibility checking only**
- not intended as a general environmental conditioning function

### Heater commands

| Function | Command |
|---|---:|
| Enable heater | `0x306D` |
| Disable heater | `0x3066` |

### Heater notes

- after reset, the heater is **disabled**
- the temperature increase depends on operating conditions and is typically only a few degrees Celsius
- heater state is reflected in the status register

## Status Register

## Reading the status register

Command:

```text
0xF32D
```

The status register is 16 bits.

## Status register bit definitions

| Bit | Field | Meaning | Default |
|---:|---|---|---|
| 15 | Alert pending status | `0`: no pending alerts, `1`: at least one pending alert | `1` |
| 14 | Reserved | keep/expect `0` | `0` |
| 13 | Heater status | `0`: heater OFF, `1`: heater ON | `0` |
| 12 | Reserved | keep/expect `0` | `0` |
| 11 | RH tracking alert | `0`: no alert, `1`: alert | `0` |
| 10 | T tracking alert | `0`: no alert, `1`: alert | `0` |
| 9:5 | Reserved | unspecified | `xxxxx` |
| 4 | System reset detected | `0`: no reset since last clear status, `1`: reset detected | `1` |
| 3:2 | Reserved | unspecified | `00` |
| 1 | Command status | `0`: last command executed successfully, `1`: last command not processed | `0` |
| 0 | Write data checksum status | `0`: last write checksum correct, `1`: last write checksum failed | `0` |

### Status register interpretation notes

- bit 4 is set when a reset was detected from:
  - hard reset
  - soft reset command
  - supply fail
- bit 1 indicates a command error such as:
  - invalid command
  - command checksum failure
- bit 0 indicates write-transfer CRC failure

## Clear status register

Command:

```text
0x3041
```

This clears flags:

- bit 15
- bit 11
- bit 10
- bit 4

## CRC-8 Checksum

Every data word is followed by an 8-bit CRC.

## CRC properties

| Property | Value |
|---|---|
| Name | CRC-8 |
| Width | 8 bit |
| Protected data | read and/or write data |
| Polynomial | `0x31` (`x^8 + x^5 + x^4 + 1`) |
| Initialization | `0xFF` |
| Reflect input | False |
| Reflect output | False |
| Final XOR | `0x00` |
| Example | `CRC(0xBEEF) = 0x92` |

## CRC scope

The CRC covers **only the two immediately preceding data bytes**.

For example:
- temperature CRC covers only the temperature word
- humidity CRC covers only the humidity word

## Raw Output Format and Conversion

Measurement data is always transferred as **16-bit unsigned integers**.

These values are already:

- calibrated
- linearized
- compensated for temperature and supply-voltage effects

## Relative humidity conversion

Use decimal arithmetic on the 16-bit raw value `SRH`:

```text
RH[%RH] = 100 * SRH / 65535
```

Equivalent form:

```text
RH[%RH] = 100 * SRH / 2^16
```

## Temperature conversion in °C

Use the 16-bit raw value `ST`:

```text
T[°C] = -45 + 175 * ST / 65535
```

Equivalent form:

```text
T[°C] = -45 + 175 * ST / 2^16
```

## Temperature conversion in °F

```text
T[°F] = -49 + 315 * ST / 65535
```

Equivalent form:

```text
T[°F] = -49 + 315 * ST / 2^16
```

### Conversion constraints

- use the raw values as **unsigned integers**
- do not apply any additional calibration or linearization beyond these formulas unless using external system-level compensation
- the formulas assume `SRH` and `ST` are used in ordinary decimal numeric form

## Recommended Driver Sequences

## Probe / initialization sequence

A robust driver should:

1. wait until power-up time has elapsed after `VDD >= VPOR`
2. optionally issue interface reset if bus state is unknown
3. read and optionally clear the status register
4. if needed, disable the heater explicitly
5. ensure the sensor is not left in periodic mode from prior operation
6. select the desired operating mode

## Single-shot measurement sequence without clock stretching

1. send one of:
   - `0x2400`
   - `0x240B`
   - `0x2416`
2. wait at least the required measurement time for the selected repeatability
3. issue an I²C read
4. read temperature word + CRC
5. read humidity word + CRC
6. validate CRCs
7. convert raw values to °C and %RH

## Single-shot measurement sequence with clock stretching

1. send one of:
   - `0x2C06`
   - `0x2C0D`
   - `0x2C10`
2. issue a read header
3. sensor may stretch the clock until data is ready
4. read temperature word + CRC
5. read humidity word + CRC
6. validate CRCs
7. convert values

## Periodic mode sequence

1. send one of the periodic measurement commands
2. wait for measurement production according to selected rate
3. send fetch command `0xE000`
4. read temperature word + CRC
5. read humidity word + CRC
6. validate CRCs
7. convert values
8. repeat fetch for future samples
9. send break command `0x3093` before changing modes or issuing unrelated commands

## ART mode sequence

1. send ART command `0x2B32`
2. use periodic fetch behavior to obtain samples
3. stop with break command `0x3093`

## Required Safety and Correctness Rules for Code Generation

Generated driver code must follow these rules:

1. **Support both legal 7-bit addresses**
   - `0x44`
   - `0x45`

2. **Do not leave `ADDR` floating**
   - it must be tied high or low

3. **Leave `ALERT` floating if unused**
   - do not drive or bias it unnecessarily

4. **Handle `nRESET` correctly**
   - active low
   - minimum pulse width 1 µs

5. **Use external pull-ups on SDA and SCL**
   - lines are open-drain

6. **Observe command spacing**
   - wait at least 1 ms after sending a command before sending another command

7. **Handle clock stretching mode correctly**
   - only available for single-shot commands that explicitly enable it

8. **Handle NACK correctly**
   - no-data-yet reads may NACK in no-clock-stretch single-shot mode and periodic fetch mode

9. **Validate CRC for read data**
   - each 16-bit word is followed by its own CRC-8

10. **Use the exact CRC parameters**
    - polynomial `0x31`
    - init `0xFF`
    - no reflection
    - final XOR `0x00`

11. **Do not assume fetch data persists after read**
    - periodic mode data memory is cleared after fetch/readout

12. **Stop periodic mode before sending unrelated commands**
    - use break command `0x3093`

13. **Treat ART as a periodic acquisition mode**
    - fetch with `0xE000`
    - stop with break

14. **Keep the I²C address stable through a transaction**
    - if ADDR is dynamically switched, it must not change between START and STOP

15. **Use the raw conversion formulas exactly**
    - humidity and temperature outputs are already compensated and linearized

16. **Handle interface reset separately from system reset**
    - interface reset does not clear the status register

17. **Be aware that general call reset affects other devices**
    - it is not sensor-specific

18. **Do not use the heater as a general operating mode**
    - it is for plausibility checking only

19. **Connect pin 7 (`R`) and center pad to VSS**
    - this is a hardware integration requirement

## Final Implementation Intent

Any autogenerated SHT30 I²C driver built from this document should:

- wire the device correctly
- support both legal addresses
- correctly implement every published command
- correctly distinguish single-shot, periodic, and ART behavior
- correctly handle NACK and clock stretching cases
- correctly validate CRC for every data word
- correctly convert raw output to humidity and temperature
- correctly manage reset and heater behavior
- avoid silent failures caused by:
  - omitted pull-ups
  - floating ADDR
  - misuse of ALERT
  - missing 1 ms command spacing
  - missing CRC validation
  - failure to stop periodic mode before reconfiguration
