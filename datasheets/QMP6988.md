# QMP6988 Barometric Pressure Sensor — Complete I²C Driver and Register Reference

## Purpose

This document is a **self-contained implementation reference** for building a complete **I²C driver** for the **QMP6988** digital barometric pressure sensor. It is intended to be sufficient on its own for:

- autogenerated driver creation
- register-level firmware implementation
- answering wiring and integration questions
- validating mode behavior, measurement flow, and compensation math

Do not assume access to any external datasheet or source material.

---

## Overview

The **QMP6988** is a digital barometric pressure and temperature sensor with an integrated **24-bit ADC**, internal **calibration coefficients stored in OTP/NVM**, and support for **I²C and SPI interfaces**.

This specification focuses on the **I²C interface**, registers, measurement procedure, compensation formulas, configuration parameters, timing, and constraints required to implement a complete driver that uses every documented feature relevant to I²C operation.

### Key capabilities

- Absolute pressure measurement: **30 kPa to 110 kPa**
- Temperature measurement
- Integrated **24-bit ADC**
- Internal **OTP calibration coefficients**
- IIR filtering
- Configurable oversampling / averaging
- Three power modes:
  - sleep
  - forced
  - normal

Typical measurement resolution:

- Pressure resolution: **0.06 Pa**
- Temperature resolution: **0.0002 °C**

### Operating limits

| Parameter | Min | Typ | Max |
|---|---:|---:|---:|
| VDD | 1.71 V | 1.8 V | 3.6 V |
| VDDIO | 1.2 V | 1.8 V | 3.6 V |
| Operating temperature | -40 °C | — | +85 °C |

### General device notes

- Pressure type: **absolute pressure**
- Medium: **air**
- Do not use with corrosive gases
- Default communication mode after reset: **I²C**
- Pressure accuracy, noise, and bandwidth depend on oversampling and IIR settings

## Electrical and Physical Interface

## Pins

| Pin | Name | I²C function / description |
|---|---|---|
| 1 | `RST` | asynchronous reset input |
| 2 | `CSB` | must be tied to **VDDIO** to select I²C mode |
| 3 | `SDI/SDO` | **SDA** |
| 4 | `SCK` | **SCL** |
| 5 | `SDO/ADDR` | I²C address select input |
| 6 | `VDDIO` | digital I/O supply |
| 7 | `GND` | ground |
| 8 | `VDD` | sensor supply |
| 9 | `VPP` | internal NVM programming pin; **leave unconnected** |

### Critical hardware constraints

- **CSB must be high (tied to VDDIO) for I²C mode**
- **Pin 9 / VPP must remain electrically unconnected**
- if the reset function is unused, **RST may be tied to GND**
- the reference connection uses **1 µF decoupling capacitors** on both **VDDIO** and **VDD**

### Digital interface characteristics relevant to I²C

- digital input low voltage max: **0.2 × VIO**
- digital input high voltage min: **0.8 × VIO**
- digital input hysteresis: **0.1 × VIO**
- digital output low voltage max in I²C mode: **0.2 × VIO** at **3 mA**
- I²C load capacitance on SDA/SCL: **up to 400 pF**

### Startup and reset timing

- power-on startup time: **10 ms**
- asynchronous reset pulse width minimum: **100 µs**

## Communication Mode

The QMP6988 supports both I²C and SPI, but this document targets **I²C-only driver implementation**.

### Mode selection rules

- **I²C mode** becomes active when **CSB = VDDIO**
- **SPI mode** becomes active when **CSB = GND**
- once CSB is pulled low, SPI mode remains selected until a reset
- after **POR** or **asynchronous reset**, the default mode is **I²C**

### I²C-only driver assumption

For all driver logic in this document, assume:

- `CSB` is permanently high
- communication is always I²C
- SPI-only fields may still appear in registers and must be preserved correctly if written

## I²C Interface

## I²C addressing

The 7-bit I²C slave address is selected by the **ADDR pin**.

| ADDR pin state | 7-bit address | 8-bit write address | 8-bit read address |
|---|---:|---:|---:|
| LOW | `0x70` | `0xE0` | `0xE1` |
| HIGH | `0x56` | `0xAC` | `0xAD` |

### Addressing rules

- the driver must support **both legal 7-bit addresses**
- the address is determined only by hardware
- prefer storing the **7-bit address** unless the MCU I²C API requires 8-bit formatted addresses

## Supported I²C speeds

- Standard Mode: **100 kbit/s**
- Fast Mode: **400 kbit/s**
- High-Speed Mode: **3.4 Mbit/s**

### I²C timing notes

The datasheet states compliance with standard, fast, and high-speed I²C timing and provides key timing values:

- SDA setup time:
  - standard / fast mode: **160 ns min**
  - HS mode, VIO = 1.62 V: **30 ns min**
  - HS mode, VIO = 1.2 V: **55 ns min**
- SDA hold time:
  - standard / fast mode, Cb ≤ 100 pF: **18–115 ns** at VIO = 1.62 V
  - standard / fast mode, Cb ≤ 400 pF: **24–150 ns** at VIO = 1.62 V
  - standard / fast mode, Cb ≤ 100 pF: **25–140 ns** at VIO = 1.2 V
  - standard / fast mode, Cb ≤ 400 pF: **45–170 ns** at VIO = 1.2 V
- SCL low pulse in HS mode:
  - **160 ns min** at VIO = 1.62 V
  - **210 ns min** at VIO = 1.2 V

Undescribed timing items are stated to follow the I²C specification.

## Register Access Semantics

### General rules

- all registers are **8-bit**
- multi-byte values are assembled from consecutive 8-bit registers
- register auto-increment is supported during reads
- if a read sequence increments beyond `0xFF`, subsequent output remains `0xFF`

### I²C write protocol

The host writes:

1. START
2. device address with write bit
3. register address
4. one or more data bytes
5. STOP

### I²C read protocol

The host reads:

1. START
2. device address with write bit
3. register address to read from
4. repeated START
5. device address with read bit
6. read one or more bytes
7. NACK on the final byte
8. STOP

## Power Modes

Power mode is configured through `CTRL_MEAS.power_mode[1:0]`.

| Bits | Mode | Behavior |
|---|---|---|
| `00` | sleep | no measurement |
| `01` | forced | single measurement, then automatic return to sleep |
| `10` | forced | same as `01` |
| `11` | normal | repeated measurements with standby intervals |

### Mode semantics

#### Sleep mode
- no measurements are performed
- I²C registers remain accessible

#### Forced mode
- a single measurement cycle is performed
- once temperature and pressure data are stored in registers, the device returns to sleep automatically

#### Normal mode
- measurements repeat continuously
- the standby interval between cycles is set by `IO_SETUP.t_standby[2:0]`

## `DEVICE_STAT` (`0xF3`, R)

Reset value: `0x00`

| Bits | Field | Meaning |
|---|---|---|
| 7:4 | reserved | read-only; ignore |
| 3 | `measure` | measurement in progress |
| 2:1 | reserved | read-only; ignore |
| 0 | `otp_update` | OTP access in progress |

### `measure`
- `0`: measurement finished or idle
- `1`: measurement in progress; data store not yet finished

### `otp_update`
- `0`: no OTP access in progress

### Driver use
- after reset, wait until `otp_update == 0` before relying on OTP coefficients
- in forced mode, polling `measure == 0` is the clean completion test

### Raw ADC output width

The device internally provides **22-bit to 24-bit effective data**, but the host always reads **24 bits**.

Valid payload behavior depends on averaging setting:

| Effective output width | Bit mapping | Note |
|---|---|---|
| 22-bit | `D21..D0` stored in bits `[23:2]`; low 2 bits are zero | averaging code `001` |
| 23-bit | `D22..D0` stored in bits `[23:1]`; low 1 bit is zero | averaging code `010` |
| 24-bit | `D23..D0` stored in bits `[23:0]` | averaging code `011` to `111` |

### Raw value assembly

The raw register contents are **unsigned 24-bit values**. The datasheet requires subtracting `2^23` to obtain the centered working values.

```text
raw_temp_u24  = (TEMP_TXD2 << 16) | (TEMP_TXD1 << 8) | TEMP_TXD0
raw_press_u24 = (PRESS_TXD2 << 16) | (PRESS_TXD1 << 8) | PRESS_TXD0

Dt = raw_temp_u24  - 2^23
Dp = raw_press_u24 - 2^23
```

### Important handling rules

- do **not** sign-extend before subtracting `2^23`
- first assemble the full 24-bit unsigned value
- then subtract `8388608`
- pressure compensation requires `Dp` and compensated `Tr`
- temperature compensation requires `Dt`

## Coefficient Reconstruction

The compensation model uses these coefficients:

- `a0`
- `a1`
- `a2`
- `b00`
- `bt1`
- `bt2`
- `bp1`
- `b11`
- `bp2`
- `b12`
- `b21`
- `bp3`

The datasheet defines them through OTP fields plus conversion factors.

## General coefficient conversion form

For the non-offset coefficients:

```text
K = A + S * OTP / 32767.0
```

Where:
- `K` is the final floating-point coefficient
- `A` is the coefficient-specific offset
- `S` is the coefficient-specific scale
- `OTP` is the signed coefficient encoding reconstructed from the OTP bytes

For the two offset-style coefficients `a0` and `b00`:

```text
K = OTP
```

The datasheet labels both as **Offset value (20Q16)**.

## Coefficient conversion parameters

| Coefficient | A | S | OTP high byte | OTP low byte |
|---|---:|---:|---|---|
| `a1`  | `-6.30E-03` | `4.30E-04` | `COE_a1_1`  | `COE_a1_0` |
| `a2`  | `-1.90E-11` | `1.20E-10` | `COE_a2_1`  | `COE_a2_0` |
| `bt1` | ` 1.00E-01` | `9.10E-02` | `COE_bt1_1` | `COE_bt1_0` |
| `bt2` | ` 1.20E-08` | `1.20E-06` | `COE_bt2_1` | `COE_bt2_0` |
| `bp1` | ` 3.30E-02` | `1.90E-02` | `COE_bp1_1` | `COE_bp1_0` |
| `b11` | ` 2.10E-07` | `1.40E-07` | `COE_b11_1` | `COE_b11_0` |
| `bp2` | `-6.30E-10` | `3.50E-10` | `COE_bp2_1` | `COE_bp2_0` |
| `b12` | ` 2.90E-13` | `7.60E-13` | `COE_b12_1` | `COE_b12_0` |
| `b21` | ` 2.10E-15` | `1.20E-14` | `COE_b21_1` | `COE_b21_0` |
| `bp3` | ` 1.30E-16` | `7.90E-17` | `COE_bp3_1` | `COE_bp3_0` |

## `a0` and `b00` packed storage

These two coefficients are stored in split fields using `COE_b00_a0_ex` plus their main bytes.

### `a0`

- upper bits: `COE_a0_1`
- middle bits: `COE_a0_0`
- extension bits: the `a0` nibble from `COE_b00_a0_ex`

Datasheet grouping:
- `COE_a0_1` contains `a0[19:12]`
- `COE_a0_0` contains `a0[11:4]`
- `COE_b00_a0_ex` contributes `a0[3:0]`

### `b00`

- upper bits: `COE_b00_1`
- middle bits: `COE_b00_0`
- extension bits: the `b00` nibble from `COE_b00_a0_ex`

Datasheet grouping:
- `COE_b00_1` contains `b00[19:12]`
- `COE_b00_0` contains `b00[11:4]`
- `COE_b00_a0_ex` contributes `b00[3:0]`

## Packed nibble split in `COE_b00_a0_ex`

The datasheet indicates this register packs:

```text
b00[3:0]  a0[3:0]
```

So the byte contains one nibble for `b00` extension bits and one nibble for `a0` extension bits.

### Reconstruction guidance

Use the datasheet’s grouping explicitly:

```text
a0_raw  = concat( a0[19:12], a0[11:4], a0[3:0] )
b00_raw = concat( b00[19:12], b00[11:4], b00[3:0] )
```

Then treat `a0` and `b00` as **20Q16 offset values**.

For the remaining coefficients:

1. combine the high and low bytes into a 16-bit OTP value
2. interpret that OTP value as the signed encoding intended by the compensation model
3. apply `K = A + S * OTP / 32767.0`

### Signedness caution

The datasheet does not fully spell out signed reconstruction details for every OTP field, but the coefficients are clearly intended for signed compensation math. Driver code should therefore treat the non-offset OTP coefficient words as signed encoded values appropriate to the formulas above.

## Compensation Equations

## Temperature compensation

```text
Tr = a0 + a1 * Dt + a2 * Dt^2
```

Where:
- `Dt` = centered raw temperature value
- `Tr` = compensated temperature in **units of 1/256 °C**

Convert to degrees Celsius with:

```text
temperature_c = Tr / 256.0
```

## Pressure compensation

```text
Pr = b00
   + bt1 * Tr
   + bp1 * Dp
   + b11 * Tr * Dp
   + bt2 * Tr^2
   + bp2 * Dp^2
   + b12 * Dp * Tr^2
   + b21 * Dp^2 * Tr
   + bp3 * Dp^3
```

Where:
- `Dp` = centered raw pressure value
- `Tr` = compensated temperature intermediate from the previous step
- `Pr` = compensated pressure in **Pa**

### Important dependency

- pressure compensation must use the already-compensated `Tr`
- do not substitute raw temperature directly into the pressure formula

## Measurement Flow

After reset or startup:

1. wait for startup completion
2. verify device identity with `CHIP_ID`
3. wait until `DEVICE_STAT.otp_update == 0`
4. read OTP coefficients
5. reconstruct all compensation coefficients
6. configure `IO_SETUP`
7. configure `IIR_CNT` if needed
8. configure `CTRL_MEAS`
9. acquire raw data
10. compute `Dt`, `Dp`, `Tr`, and `Pr`

## Driver Initialization Sequence

A correct driver should follow this order after power-on or reset:

1. wait at least **10 ms**
2. read `CHIP_ID`
3. verify `CHIP_ID == 0x5C`
4. ensure `DEVICE_STAT.otp_update == 0`
5. read OTP coefficient registers once
6. reconstruct all compensation coefficients and cache them
7. configure `IO_SETUP`
8. configure `IIR_CNT` if desired
9. configure `CTRL_MEAS`
10. enter sleep, forced, or normal mode as required

### Coefficient cache recommendation

Store all reconstructed coefficients in driver state and avoid rereading OTP on every sample.

---

## Forced Mode Sequence

Use forced mode for on-demand conversions.

1. configure oversampling in `CTRL_MEAS`
2. write `power_mode = forced` (`01` or `10`)
3. poll `DEVICE_STAT.measure`
4. continue when `measure == 0`
5. read:
   - `TEMP_TXD2..0`
   - `PRESS_TXD2..0`
6. compute:
   - `Dt`
   - `Dp`
   - `Tr`
   - `Pr`
7. return compensated results

### Forced mode note

After the conversion completes, the device returns to **sleep mode automatically**.

---

## Normal Mode Sequence

Use normal mode for repeated measurements.

1. configure standby in `IO_SETUP`
2. configure oversampling in `CTRL_MEAS`
3. set `power_mode = 11`
4. read the latest pressure and temperature values periodically

### Normal mode note

Measurement and standby periods repeat automatically. The driver must read data often enough for the intended effective data rate.

## Status and Polling Behavior

Use `DEVICE_STAT` for synchronization.

### OTP activity
- `otp_update == 1` means OTP access is active
- do not rely on coefficients until this clears

### Measurement activity
- `measure == 1` means a measurement is still in progress
- `measure == 0` means measurement is complete or the device is idle

## Reset Behavior

The QMP6988 supports both **software reset** and **asynchronous reset**.

## Software reset

Write:

```text
RESET = 0xE6
```

This triggers a reset.

## Asynchronous reset pin sequence

1. drive `RST` high for at least **100 µs**
2. drive `RST` low
3. wait **10 ms**

### After any reset

- wait startup time
- recheck `CHIP_ID`
- reread OTP coefficients if driver state was lost or reset handling requires it
- restore configuration registers
- do not assume continuous-mode state survives reset

## Required Safety and Correctness Rules for Code Generation

Generated driver code must follow these rules:

1. **Use I²C mode only**
   - `CSB` high
   - address selected by ADDR pin

2. **Preserve reserved bits**
   - `IO_SETUP[4:3] = 0`
   - `IO_SETUP[1] = 0`
   - `I2C_SET[7:3] = 0`
   - `IIR_CNT[7:3] = 0`

3. **Validate chip identity**
   - `CHIP_ID` must equal `0x5C`

4. **Wait for startup/reset completion**
   - at least 10 ms after power-up or reset

5. **Handle OTP readiness**
   - do not rely on coefficients while `DEVICE_STAT.otp_update == 1`

6. **Read OTP coefficients once per reset cycle**
   - cache them in driver state

7. **Assemble raw ADC values correctly**
   - read 24-bit unsigned
   - subtract `2^23`

8. **Use temperature compensation before pressure compensation**
   - compute `Tr` first
   - compute `Pr` second

9. **Do not rewrite `IIR_CNT` unnecessarily**
   - writing it resets filter state

10. **Forced mode returns to sleep automatically**
    - do not assume the device remains in forced mode

11. **Do not connect or use VPP**
    - pin 9 must stay unconnected

12. **Do not skip temperature when pressure is required**
    - pressure compensation depends on `Tr`

13. **Treat `I2C_SET` reset default as ambiguous**
    - only write intended bits
    - preserve reserved bits

14. **Preserve explicit coefficient field structure**
    - especially `COE_b00_a0_ex` split fields
